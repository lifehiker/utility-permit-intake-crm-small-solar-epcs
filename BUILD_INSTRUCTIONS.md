# Utility and Permit Intake CRM for Small Solar EPCs

## Overview

**Web SaaS:** Utility and Permit Intake CRM for Small Solar EPCs is a branded customer portal and internal workflow CRM for small residential and light-commercial solar installers. It solves the operational bottleneck where utility bills, 12-month usage history, interval data, rate schedules, authorization forms, site photos, AHJ permit documents, and interconnection steps are currently scattered across email threads, PDFs, spreadsheets, and shared drives. The MVP gives each installer a branded intake portal for customers and an internal project tracker so sales, design, and ops teams can see exactly what is missing before proposal, permitting, and utility interconnection work begins.

## Why this will succeed

### Comparable products that are working

1. **UtilityAPI** — https://utilityapi.com  
   - **Known size:** Established utility data authorization and interval data access provider; public customer/user counts were not provided in the research.  
   - **What works:** UtilityAPI proves that solar-adjacent teams pay for structured utility data access and customer authorization workflows.  
   - **Gap we exploit:** UtilityAPI is focused on utility data access, not a lightweight end-to-end intake CRM for small EPCs managing bills, site documents, AHJ permit documents, interconnection checklists, and customer follow-ups.

2. **OpenSolar** — https://www.opensolar.com  
   - **Known size:** Large solar design/proposal platform with global presence; exact user/customer count was not provided in the research.  
   - **What works:** OpenSolar validates that installers use specialized solar SaaS for proposal and design workflows.  
   - **Gap we exploit:** OpenSolar is primarily positioned around design, proposals, sales, hardware, and finance workflows. Small EPCs may still manage pre-proposal utility bills, authorization forms, site documents, and follow-up tasks outside the platform.

3. **Enerflo** — https://www.enerflo.com  
   - **Known size:** Established residential solar sales and operations platform; pricing and customer count were not provided in the research.  
   - **What works:** Enerflo validates demand for solar-specific sales and operations platforms.  
   - **Gap we exploit:** Enerflo is a broader solar platform. This product is intentionally narrower: intake, document collection, permit/interconnection status tracking, and branded customer follow-up for small regional EPCs that do not want to adopt a heavier platform.

### Why this specific opportunity is different / better

This product should **not** compete as a full solar design, proposal, financing, or enterprise operations platform. The differentiation is delivery and scope:

## Target user

They typically manage 5-50 active prospects/projects across proposal, site review, permitting, and utility interconnection. They currently collect customer utility bills, 12-month usage history, interval data, rate schedules, authorization forms, site photos, roof/electrical documents, AHJ forms, and utility interconnection paperwork through email, PDFs, Google Drive folders, spreadsheets, and manual reminders.

Current options fail them because:

- Solar design tools focus on proposal/design, not intake completeness.
- Utility data tools focus on authorization/data access, not full project document collection.
- Generic CRMs are not built around utility bills, usage data, AHJ permits, and interconnection steps.
- Spreadsheets do not create a professional customer-facing intake experience.
- Email threads hide what is missing and who needs to follow up.

Where they already hang out online:

- NABCEP directories and professional ecosystem

- Solar Power World contractor ecosystem
- r/solarbusiness

## Core features (MVP)

- **Organization signup and onboarding**
  - Create company workspace.
  - Add company name, logo, brand color, contact email, and phone.
  - Generate a branded portal URL.

- **Team authentication**
  - Google OAuth login via NextAuth.
  - Organization-level membership.
  - Roles: owner, admin, member.

- **Customer/project creation**
  - Internal user creates a project with:
    - Customer name
    - Customer email
    - Phone
    - Address
    - Project type: residential, commercial, light-commercial
    - Current stage
    - Assigned internal owner

- **Branded customer intake portal**
  - Public magic-link page for each project.
  - Customer can upload requested documents without creating an account.
  - Portal shows required items, completed items, and instructions.
  - Portal uses EPC logo/brand color.

- **Document request checklist**
  - Default required items:
    - Recent utility bill
    - 12-month usage history
    - Interval data file
    - Current rate schedule
    - Utility account number
    - Authorization form
    - Site photos
    - Roof/electrical photos
    - AHJ documents
    - Interconnection documents
  - Internal user can mark each item required/not required per project.
  - Internal user can add custom document requests.

- **File uploads**
  - Upload PDFs, images, CSVs, XLSX files.
  - Store files locally/S3-compatible storage depending deployment configuration.
  - Show upload date, uploader name/email, and file type.
  - Allow internal users to download files.

- **Project status board**
  - Kanban or table view with stages:
    - New intake
    - Waiting on customer
    - Ready for proposal
    - Permit prep
    - AHJ submitted
    - Utility interconnection
    - Complete
  - Filter by stage, assigned owner, missing documents, and project type.

- **Missing-item tracker**
  - Each project displays percentage complete.
  - “Missing items” panel lists incomplete required documents.
  - Dashboard shows projects blocked by missing customer documents.

- **Email reminders**
  - Send portal invite to customer.
  - Send manual reminder for missing items.
  - Include branded link to the intake portal.
  - Track last reminder sent timestamp.

- **Internal notes and activity log**
  - Add notes to a project.
  - Log uploads, status changes, reminders, and checklist changes.

- **Templates**
  - Built-in templates:
    - Residential solar proposal intake
    - Commercial solar proposal intake
    - AHJ permit checklist
    - Utility interconnection checklist
  - Apply template when creating a project.

- **Subscription billing**
  - Stripe checkout.
  - Active subscription required after trial.
  - Enforce plan limits by number of active projects and team members.

- **Basic admin/settings**
  - Company branding.
  - Team member invites.
  - Billing portal link.
  - Email sender settings placeholder.

## Key Implementation Notes

### Database (only if PRD requires persistence)
- Use **SQLite** via Prisma: `provider = "sqlite"`, `url = env("DATABASE_URL")`
- `DATABASE_URL` defaults to `file:/data/app.db` in the Dockerfile — no Coolify config needed
- For local dev: `DATABASE_URL="file:./dev.db"` in `.env.local`
- Run `npx prisma db push` after schema changes
- **Do NOT use PostgreSQL** — no PostgreSQL server is provisioned in the deployment environment

### Authentication (only if PRD requires user login)
- Use **NextAuth v5** with the **Credentials provider** (email + password)
- **Do NOT use Google OAuth** — no OAuth credentials are configured in deployment
- Hash passwords with **bcrypt** (`bcryptjs` package)
- Session strategy: **`"jwt"`** (not `"database"`) — required for Credentials provider

### Public Routes Contract (CRITICAL when adding auth)

If you add auth, the following routes MUST render to status 200 for an unauthenticated visitor — they are how customers find the product, evaluate it, and sign up:

- `/` — homepage / landing
- `/pricing` — pricing page (if pricing exists)
- `/blog`, `/blog/[slug]` — blog (if any)
- `/about`, `/contact`, `/legal/*` — marketing supporting pages
- `/login`, `/signup` — auth entry points themselves

Implementation rules:
- Place public pages OUTSIDE any `(app)` / `(authenticated)` / `(dashboard)` route group, AND outside any layout that calls `requireUser()` / `auth()` / `getServerSession()`.
- Authenticated routes go INSIDE a route group whose layout gates with `requireUser()` (e.g. `src/app/(app)/dashboard`, `src/app/(app)/settings`, `src/app/(app)/billing` if billing is the user's own subscription page). The verifier knows about Next.js route groups and expects these to redirect to `/login` when visited anonymously.
- If you write a `middleware.ts`, its `matcher` MUST exclude the public routes above. A blanket matcher like `/((?!api|_next).*)` that protects everything is a build failure.
- A user with no session who visits `/` and lands on `/login` is a build failure — Forge's verifier will detect it and reject the build.

If the PRD does NOT require auth, do NOT add NextAuth at all. No middleware, no `requireUser`, no `(app)` group — just public pages.
- `AUTH_SECRET` defaults are baked into the Dockerfile; app works out of the box

### Payments (only if PRD explicitly requires it)
- Use Stripe only when the PRD's core functionality requires payment processing
- If adding Stripe: handle `checkout.session.completed`, `customer.subscription.updated`, `customer.subscription.deleted` at `/api/webhooks/stripe`
- Make the app functional without Stripe credentials (show upgrade prompts rather than crashing)

### Email (only if PRD explicitly requires it)
- Use Resend only when the PRD requires transactional email
- **CRITICAL: NEVER instantiate `new Resend(...)` at module level** — always inside the handler function body with a guard:
  ```ts
  // WRONG — causes build failure (next build evaluates module-level code):
  const resend = new Resend(process.env.RESEND_API_KEY);
  export async function POST(req) { ... }

  // CORRECT — lazy init inside handler:
  export async function POST(req) {
    if (!process.env.RESEND_API_KEY) {
      console.log("[email] RESEND_API_KEY not set, skipping email");
      return NextResponse.json({ ok: true });
    }
    const { Resend } = await import("resend");
    const resend = new Resend(process.env.RESEND_API_KEY);
    ...
  }
  ```
- Next.js evaluates module-level code during `next build` page data collection — missing env vars at module scope crash the build
- Same rule applies to Stripe, OpenAI, and ALL third-party SDK clients — always lazy-initialize inside handler functions

### Fonts
- **Do NOT use `next/font/google`** — it fetches font files during `next build`, which can hang or fail in automated/offline build environments.
- Use CSS/system font stacks by default.
- If a custom font is truly required, commit the font file into the repo and use `next/font/local`; never depend on a build-time network font fetch.

### Deployment
- Coolify with Docker — always set `output: "standalone"` in next.config.ts
- Do NOT add `COPY public ./public` in the Dockerfile if the public/ directory is empty or doesn't exist
- The Dockerfile's CMD initializes the SQLite database on first start via `prisma db push`
- App must respond healthy on port 3000 within 60 seconds of container start
- **CRITICAL Dockerfile rule**: In the `deps` stage, always use `RUN npm ci --ignore-scripts`. Without `--ignore-scripts`, the `postinstall` hook (e.g. `prisma generate`) runs before `prisma/schema.prisma` is copied, causing the build to fail. Run `npx prisma generate` explicitly in the `builder` stage after `COPY . .`.
- **Do NOT run `npx prisma db push` in the Dockerfile builder stage** — it is not needed at build time. The CMD already runs `prisma db push` at container startup when a real `/data/app.db` path is available.
- **Use `node:20-slim` (not `node:20-alpine`)** — Debian slim avoids Alpine musl/libssl issues. Add `RUN apt-get update -y && apt-get install -y openssl && rm -rf /var/lib/apt/lists/*` to BOTH the builder and runner stages for Prisma's schema engine.
- **Set `binaryTargets` in schema.prisma** — always include `binaryTargets = ["native", "debian-openssl-3.0.x"]` in the Prisma generator block so Prisma generates the correct engine binary for the Debian container environment.
- **Copy full node_modules in the runner stage** — do NOT selectively copy `node_modules/.prisma`, `node_modules/prisma`, `node_modules/@prisma` separately. Instead use `COPY --from=builder --chown=nextjs:nodejs /app/node_modules ./node_modules` to copy everything. Prisma v6+ CLI has deep transitive deps (`@prisma/config` → `effect`, `c12`, etc.) that break if not all present.
